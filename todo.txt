Current screens in the donor app:

Home (/ / HomePage)

Purpose: Landing page for donors; explains the campaign and steers them toward making a gift.
Key elements: centered hero card with “COP Giving Mobile” headline, subtitle, supporting sentence, and a primary “Give now” button that navigates to /branches.
Branch selector (/branches / BranchSelectPage)

Purpose: Lets donors search for and pick a donation-ready branch before starting checkout.
Key elements: transparent Ionic header (“Choose a donation branch”), a raised search bar, loading/error/empty states (cards with spinner or retry button), and a list of branch cards showing name, district, area, and branch code. Tapping a branch saves it to SelectedBranchService and goes to /donate.
Donation form (/donate / DonatePage)

Purpose: Collects donation details (branch, category, amount, optional email) and triggers the backend checkout.
Key elements: Selected branch card, reactive form with labeled ion-select (categories), ion-input for amount and email, validation messaging, primary “Give now” button that posts to /api/donations/checkout/, stores the summary in both state and sessionStorage, and redirects the browser to Stripe.
Success confirmation (/donate/success / DonateSuccessPage)

Purpose: Shows verified donation details after Stripe redirects back; verifies session with /api/donations/verify-checkout-session/.
Key elements: Thank-you card with summary copy, optional backend-verified status line, detail rows for branch, category, amount, reference, email, session ID, “Give again” and “Back home” buttons, and loading/error states tied to the verification call. Falls back to the stored summary if verification can’t run.
Cancel/reset (/donate/cancel / DonateCancelPage)

Purpose: Informs the donor when they cancel/abort the Stripe flow; optionally shows what they were about to give.
Key elements: Card confirming no payment went through, optional branch/category/amount (from stored summary), primary “Try again” button linking to /branches, and secondary “Back home” button.
Each screen is a standalone Ionic component with its own template/styles, uses shared services (SelectedBranchService, DonationFlowStateService, DonationsService), and routes are wired through AppRoutingModule.

Add Apple Pay / Google Pay (one config change away)
Thought for 8s

Yes — and it’s a bit more than one config change.

For PaymentSheet to show Apple Pay and Google Pay, you need both code config and native platform setup.

Apple Pay needs:

an Apple Merchant ID
an Apple Pay certificate in Stripe
the Apple Pay capability enabled in Xcode
PaymentSheet configured with applePay using your merchant ID and business country code.

Google Pay needs:

Google Pay enabled in your Android app manifest with com.google.android.gms.wallet.api.enabled=true
Stripe Android/PaymentSheet setup on Android
a device/account with Google Pay available for testing.

Also, Stripe’s mobile docs make clear that PaymentSheet is the recommended in-app UI and can surface Apple Pay and Google Pay when configured.

What to do next

For Apple Pay:

Create a Merchant ID in Apple Developer.
In Stripe Dashboard, generate the Apple Pay certificate for that Merchant ID.
In Xcode, add Signing & Capabilities → Apple Pay and select that Merchant ID.
In your PaymentSheet config, pass:
merchantId
merchantCountryCode

For Google Pay:

Add this inside the Android <application> tag:
<meta-data
  android:name="com.google.android.gms.wallet.api.enabled"
  android:value="true" />
Make sure your Android setup supports Google Pay and your test device has a payment method configured.
In your app code

Your StripePaymentService will need PaymentSheet config extended to include:

Apple Pay config
Google Pay config
likely merchant display name / country code values from environment
Recommended rollout

Do Google Pay first on Android, because it’s usually faster to validate.
Then do Apple Pay on iPhone once the Merchant ID/certificate/capability are in place.

Brief summary

Keeping Stripe Connect is the right call for your current architecture. With one branch = one connected account, branch selection is not just reporting metadata anymore; it directly affects payout routing and donation eligibility, so you should keep branch selection in the donor flow, but make it faster and smarter rather than removing it. Your docs also show the current flow is already production-ready end to end, including branch-specific routing, native/mobile payment support, webhook reconciliation, and verified success screens.

So the best app changes are:

Do not remove branch selection
Because funds are routed per branch account, the donor must still choose a branch before payment. Your current flow and payment architecture depend on that.
Make branch selection faster
Keep /branches, but add:
“Recent branches”
“My branch” pinned at top
auto-select last used branch on return visits
district/area shortcuts
This keeps Stripe Connect intact while reducing friction. Your current branch step already has search and clean cards, so this is an enhancement, not a rewrite.
Add Quick Give after branch is known
Since branch is required for routing, Quick Give should work like:
Home shows “Give again to Milan Central”
tap goes straight to /donate with last branch preselected
user can still change branch
This is better than skipping branch universally.
Keep donate as the main optimization point
Your donate screen already has category chips, preset amounts, custom amount, optional email, and recurring toggle stub. That means the biggest win is not a new screen, but smarter defaults:
remember last category
remember last amount
prefill donor email if previously used
show category-specific presets
move recurring from stub to working feature later

Upgrade the success screen into a re-donation loop
You already have verified summary + “Give again”. Make “Give again” actually reuse:
branch
category
amount
That creates a much stronger repeat-donation path.
Add branch availability states in donor app
Since donations must only go to onboarded, active branches, donor-facing branch cards should visibly show:
Available
Temporarily unavailable
Setup in progress
Your onboarding model already depends on donation-enabled status, so exposing that state in the app will reduce failed attempts and confusion.
Do not merge away branch identity
Because Stripe Connect needs branch-level readiness and branch-level routing, the app should reinforce trust by showing the selected branch clearly on donate and success screens. Your current UI already does that well, so preserve it.

My recommended priority order is:

P1: recent branch + last-used branch preselection
P2: quick give from home using remembered branch
P3: smarter donate defaults
P4: donor-visible branch availability state
P5: make recurring actually work
